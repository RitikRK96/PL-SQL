What is SQL?

The language databases speak
🏪 Real-World Analogy Think of a database as a massive filing cabinet in a company. Each drawer is a table, each folder is a row, and the label on the folder is a column. SQL is your language to tell the filing assistant what to find, add, change, or remove.

SQL vs PL/SQL — What's the difference?

SQLPL/SQL
Structured Query LanguageProcedural Language / SQL
Single statementsBlocks of code with logic
Only works with dataHas IF/ELSE, loops, variables
Declarative: "what to get"Procedural: "how to process"
Run one at a timeSend whole programs to DB

SQL Command Categories

  • DDL Data Definition Language — CREATE, ALTER, DROP, TRUNCATE
  • DML Data Manipulation Language — SELECT, INSERT, UPDATE, DELETE
  • DCL Data Control Language — GRANT, REVOKE
  • TCL Transaction Control Language — COMMIT, ROLLBACK, SAVEPOINT
1.2

DDL & DML

Creating tables and working with data

CREATE TABLE — Building your structure

A table is like a spreadsheet template — you define the columns once, then fill in rows of data.

SQL — Create & Populate Table
-- Create an employees table (like designing a form)
CREATE TABLE employees (
    emp_id     NUMBER(6)       PRIMARY KEY,   -- Unique identifier
    first_name VARCHAR2(50)  NOT NULL,      -- Text, max 50 chars
    last_name  VARCHAR2(50)  NOT NULL,
    salary     NUMBER(10,2),               -- Decimal number
    dept_id    NUMBER(4),
    hire_date  DATE DEFAULT SYSDATE         -- Defaults to today
);

-- Insert rows (like filling in the form)
INSERT INTO employees (emp_id, first_name, last_name, salary, dept_id)
VALUES (1, 'Amit', 'Sharma', 75000, 10);

INSERT INTO employees (emp_id, first_name, last_name, salary, dept_id)
VALUES (2, 'Priya', 'Verma', 90000, 20);

INSERT INTO employees (emp_id, first_name, last_name, salary, dept_id)
VALUES (3, 'Rahul', 'Nair', 60000, 10);

-- Update a value
UPDATE employees
SET salary = 80000
WHERE emp_id = 3;

-- Delete a record
DELETE FROM employees
WHERE emp_id = 3;

-- Always commit to save DML changes!
COMMIT;

-- Undo changes (before COMMIT)
ROLLBACK;
💡 Developer Tip Always have a WHERE clause in UPDATE and DELETE. Without it, you'll modify every single row. "DELETE FROM employees" wipes the entire table! Test with a SELECT first using the same WHERE condition.
⚠️ Warning DDL commands (CREATE, DROP, ALTER) auto-commit. You cannot ROLLBACK them. DML (INSERT, UPDATE, DELETE) must be explicitly committed.

SELECT Queries

The most important SQL statement you'll ever learn
📚 Analogy SELECT is like asking a librarian for books. You say: "Give me (SELECT) the title and author (columns) from the library catalog (FROM table) where the genre is Sci-Fi (WHERE condition) sorted by year (ORDER BY)."
SQL — SELECT Deep Dive
-- 1. Basic SELECT
SELECT first_name, last_name, salary
FROM   employees;

-- 2. SELECT with WHERE
SELECT first_name, salary
FROM   employees
WHERE  salary > 70000;

-- 3. Multiple conditions
SELECT *
FROM   employees
WHERE  dept_id = 10
  AND  salary BETWEEN 50000 AND 90000;

-- 4. LIKE for pattern matching
SELECT *
FROM   employees
WHERE  first_name LIKE 'A%';   -- Names starting with A

-- 5. ORDER BY
SELECT first_name, salary
FROM   employees
ORDER BY salary DESC;        -- Highest first

-- 6. Column alias (rename output)
SELECT first_name AS "Employee Name",
       salary * 12 AS annual_salary
FROM   employees;

-- 7. DISTINCT (remove duplicates)
SELECT DISTINCT dept_id
FROM   employees;

-- 8. NULL handling
SELECT first_name, NVL(salary, 0) AS salary   -- Replace NULL with 0
FROM   employees;

-- 9. IN operator
SELECT *
FROM   employees
WHERE  dept_id IN (10, 20, 30);

JOINs — Connecting Tables

The most important concept in relational databases
🔗 Analogy Imagine you have two lists: one is a list of employees (with dept_id) and another is a list of departments (with dept_id and dept_name). A JOIN is like a "zip" operation — it connects the two lists wherever dept_id matches, so you can see each employee's department name.
SQL — All JOIN Types
-- INNER JOIN: Only rows with matches in BOTH tables
SELECT e.first_name, e.salary, d.dept_name
FROM   employees   e
JOIN   departments d ON e.dept_id = d.dept_id;

-- LEFT OUTER JOIN: All employees, even if no dept match
SELECT e.first_name, d.dept_name
FROM   employees   e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
-- Employees with no department get NULL for dept_name

-- RIGHT OUTER JOIN: All departments, even with no employees
SELECT e.first_name, d.dept_name
FROM   employees   e
RIGHT JOIN departments d ON e.dept_id = d.dept_id;

-- FULL OUTER JOIN: Everything from both tables
SELECT e.first_name, d.dept_name
FROM   employees   e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;

-- SELF JOIN: Employee and their Manager (same table!)
SELECT e.first_name AS employee,
       m.first_name AS manager
FROM   employees e
JOIN   employees m ON e.manager_id = m.emp_id;

JOIN Quick Reference

JOIN TypeReturnsUnmatched rows?
INNER JOINOnly matching rows❌ Excluded
LEFT JOINAll left + matching rightLeft: ✅ NULLs for right
RIGHT JOINAll right + matching leftRight: ✅ NULLs for left
FULL OUTER JOINEverything from both✅ NULLs both sides
CROSS JOINEvery combo (cartesian)N/A

Interview Q Classic Question

"What is the difference between INNER JOIN and LEFT JOIN?"

▶ Show Answer
INNER JOIN returns only rows where there is a match in both tables. If an employee has no department, they are excluded.

LEFT JOIN returns all rows from the left table (employees) plus matching rows from the right table. Employees without departments appear with NULL values for department columns. Use LEFT JOIN when you don't want to lose records from the primary (left) table.

GROUP BY & Aggregate Functions

Summarizing and analyzing data
SQL — Aggregations
-- Aggregate functions
SELECT
    COUNT(*),               -- Total number of rows
    COUNT(salary),          -- Rows where salary is NOT NULL
    SUM(salary),            -- Total salary
    AVG(salary),            -- Average salary
    MAX(salary),            -- Highest salary
    MIN(salary)             -- Lowest salary
FROM employees;

-- GROUP BY: Summarize per department
SELECT   dept_id,
         COUNT(*) AS emp_count,
         AVG(salary) AS avg_salary,
         MAX(salary) AS max_salary
FROM     employees
GROUP BY dept_id
ORDER BY avg_salary DESC;

-- HAVING: Filter groups (like WHERE but for aggregates)
SELECT   dept_id,
         AVG(salary) AS avg_sal
FROM     employees
GROUP BY dept_id
HAVING   AVG(salary) > 70000;   -- Only depts where avg > 70k

-- ROLLUP: Subtotals + grand total
SELECT   dept_id, SUM(salary)
FROM     employees
GROUP BY ROLLUP(dept_id);
💡 Key Rule Every column in SELECT must either be inside an aggregate function (SUM, COUNT, etc.) OR listed in the GROUP BY clause. This is one of the most common beginner mistakes.

Subqueries & CTEs

Queries inside queries — the power tool
SQL — Subqueries & WITH clause
-- Scalar subquery: returns a single value
SELECT first_name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees);

-- Subquery in FROM clause (inline view)
SELECT dept_id, avg_sal
FROM   (
    SELECT   dept_id, AVG(salary) AS avg_sal
    FROM     employees
    GROUP BY dept_id
) dept_stats
WHERE avg_sal > 65000;

-- EXISTS: Check if a related record exists
SELECT dept_name
FROM   departments d
WHERE  EXISTS (
    SELECT 1
    FROM   employees e
    WHERE  e.dept_id = d.dept_id
);

-- CTE (Common Table Expression) — Cleaner than subqueries
WITH high_earners AS (
    SELECT emp_id, first_name, salary, dept_id
    FROM   employees
    WHERE  salary > 80000
)
SELECT h.first_name, d.dept_name
FROM   high_earners h
JOIN   departments  d ON h.dept_id = d.dept_id;

What is PL/SQL?

Oracle's procedural extension to SQL
🏭 Analogy If SQL is like sending individual orders to a factory ("make this", "ship that"), PL/SQL is like giving the factory a complete production manual — "if inventory is low, order more; if the order is urgent, prioritize it; for each item in the list, do this process." It's the difference between one instruction and a whole program.

Why PL/SQL over plain SQL?

  • ✅ Add conditional logic (IF/THEN/ELSE) to database operations
  • ✅ Loop over result sets without fetching to application layer
  • ✅ Handle errors gracefully with exception blocks
  • ✅ Bundle logic into reusable stored procedures & functions
  • ✅ Dramatically reduces network round-trips (performance)
  • ✅ Business logic lives in the database — secure and centralized
2.2

PL/SQL Block Structure

The DNA of every PL/SQL program
📋 Analogy A PL/SQL block is like a recipe card. It has: a DECLARE section (list your ingredients), a BEGIN section (the actual cooking steps), and an EXCEPTION section (what to do if something goes wrong — e.g., "if we're out of eggs, substitute with flax").
PL/SQL — Block Structure
/*  PL/SQL Block Anatomy
    ┌─────────────────────────────────┐
    │  DECLARE  (optional)            │  ← Variables, constants, types
    │  BEGIN    (mandatory)           │  ← Your logic goes here
    │  EXCEPTION (optional)           │  ← Error handling
    │  END;     (mandatory)           │
    └─────────────────────────────────┘ */

DECLARE
    -- Variable declarations
    v_emp_name   VARCHAR2(100);
    v_salary     NUMBER(10, 2) := 0;   -- := is assignment in PL/SQL
    v_message    VARCHAR2(200) := 'Hello, PL/SQL!';
    c_max_sal    CONSTANT NUMBER := 200000;  -- Cannot be changed

BEGIN
    -- Retrieve a value from the DB into a variable
    SELECT first_name || ' ' || last_name, salary
    INTO   v_emp_name, v_salary
    FROM   employees
    WHERE  emp_id = 1;

    -- Print output (visible in SQL Developer output pane)
    DBMS_OUTPUT.PUT_LINE('Employee: ' || v_emp_name);
    DBMS_OUTPUT.PUT_LINE('Salary:   ' || v_salary);

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('No employee found!');
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
END;
/  -- The / runs the block in SQL*Plus / SQL Developer
💡 Enable Output Before running PL/SQL blocks, run: SET SERVEROUTPUT ON; — otherwise DBMS_OUTPUT.PUT_LINE won't show anything!

SELECT INTO — The core way to read data in PL/SQL

In PL/SQL, you can't just SELECT freely. You must store the result in variables using INTO. This requires the query to return exactly one row — if it returns zero or more than one, Oracle throws an exception.

Variables, Types & %TYPE / %ROWTYPE

Anchored declarations — the smart way to declare
PL/SQL — Variables & Anchored Types
DECLARE
    -- Regular declarations
    v_name    VARCHAR2(50)   := 'Oracle';
    v_count   NUMBER          := 0;
    v_flag    BOOLEAN         := TRUE;

    -- %TYPE: Variable inherits the exact type of a table column
    -- ADVANTAGE: If column type changes, your code auto-adapts!
    v_sal     employees.salary%TYPE;
    v_fname   employees.first_name%TYPE;

    -- %ROWTYPE: Variable holds an entire table row
    v_emp     employees%ROWTYPE;

BEGIN
    -- Using %TYPE
    SELECT first_name, salary
    INTO   v_fname, v_sal
    FROM   employees
    WHERE  emp_id = 1;

    DBMS_OUTPUT.PUT_LINE(v_fname || ' earns ' || v_sal);

    -- Using %ROWTYPE (fetch entire row)
    SELECT * INTO v_emp
    FROM   employees
    WHERE  emp_id = 1;

    -- Access individual columns using dot notation
    DBMS_OUTPUT.PUT_LINE(v_emp.first_name || ' - Dept: ' || v_emp.dept_id);

END;
/
💡 Best Practice Always prefer %TYPE and %ROWTYPE over hardcoded data types. If the DBA changes a column from VARCHAR2(50) to VARCHAR2(100), your code won't break and won't need to be redeployed.
2.4

Control Structures

IF/THEN/ELSE and CASE — making decisions in code
PL/SQL — IF/ELSIF/ELSE and CASE
DECLARE
    v_salary  employees.salary%TYPE;
    v_grade   VARCHAR2(10);
    v_emp_id  NUMBER := 1;

BEGIN
    SELECT salary INTO v_salary
    FROM   employees
    WHERE  emp_id = v_emp_id;

    -- IF / ELSIF / ELSE
    IF    v_salary >= 100000 THEN
        v_grade := 'A';
    ELSIF v_salary >= 70000  THEN
        v_grade := 'B';
    ELSIF v_salary >= 50000  THEN
        v_grade := 'C';
    ELSE
        v_grade := 'D';
    END IF;

    DBMS_OUTPUT.PUT_LINE('Salary Grade: ' || v_grade);

    -- CASE expression (cleaner for multiple values)
    v_grade := CASE
        WHEN v_salary >= 100000 THEN 'A - Executive'
        WHEN v_salary >= 70000  THEN 'B - Senior'
        WHEN v_salary >= 50000  THEN 'C - Mid-level'
        ELSE                          'D - Junior'
    END;

    DBMS_OUTPUT.PUT_LINE('Level: ' || v_grade);

END;
/

2.5

Loops

LOOP, WHILE, FOR — iterating in PL/SQL
PL/SQL — All Loop Types
DECLARE
    v_count  NUMBER := 1;
    v_sum    NUMBER := 0;

BEGIN
    -- 1. Basic LOOP (manually EXIT)
    LOOP
        v_sum := v_sum + v_count;
        v_count := v_count + 1;
        EXIT WHEN v_count > 10;    -- Must have an exit condition!
    END LOOP;
    DBMS_OUTPUT.PUT_LINE('Sum 1-10 = ' || v_sum);

    -- 2. WHILE LOOP (check condition before each iteration)
    v_count := 1;
    v_sum   := 0;
    WHILE v_count <= 10 LOOP
        v_sum   := v_sum + v_count;
        v_count := v_count + 1;
    END LOOP;

    -- 3. FOR LOOP (fixed range — most common)
    v_sum := 0;
    FOR i IN 1..10 LOOP      -- i is auto-declared, no DECLARE needed
        v_sum := v_sum + i;
    END LOOP;

    -- 4. REVERSE FOR LOOP
    FOR i IN REVERSE 1..5 LOOP
        DBMS_OUTPUT.PUT_LINE('Count: ' || i);   -- Prints 5,4,3,2,1
    END LOOP;

END;
/

3.1

Cursors

Processing multiple rows one at a time
☕ Analogy A cursor is like a reading pointer on a spreadsheet. Your SELECT query returns many rows, and the cursor lets you point to one row at a time, read it, process it, then move to the next. Think of it like reading a book page by page, rather than trying to read all pages at once.

Implicit vs Explicit Cursors

  • Implicit Cursor: Auto-created by Oracle for any DML statement (INSERT/UPDATE/DELETE/SELECT INTO). Access via SQL%ROWCOUNT, SQL%FOUND, SQL%NOTFOUND
  • Explicit Cursor: Manually declared and controlled when you need to process multiple rows
PL/SQL — Cursors (All Styles)
-- ─── STYLE 1: Cursor FOR Loop (RECOMMENDED — simplest) ───
BEGIN
    FOR emp_rec IN (SELECT first_name, salary, dept_id
                   FROM   employees
                   WHERE  dept_id = 10)
    LOOP
        DBMS_OUTPUT.PUT_LINE(
            emp_rec.first_name || ' → $' || emp_rec.salary
        );
    END LOOP;  -- Cursor auto-opened and closed!
END;
/

-- ─── STYLE 2: Explicit Cursor (full control) ───
DECLARE
    -- Step 1: Declare the cursor
    CURSOR c_emp IS
        SELECT emp_id, first_name, salary
        FROM   employees
        WHERE  salary > 60000
        ORDER BY salary DESC;

    v_emp  c_emp%ROWTYPE;   -- Row variable matching cursor columns

BEGIN
    OPEN  c_emp;               -- Step 2: Execute the query
    LOOP
        FETCH c_emp INTO v_emp; -- Step 3: Get next row
        EXIT WHEN c_emp%NOTFOUND;  -- Stop when no more rows

        DBMS_OUTPUT.PUT_LINE(
            v_emp.first_name || ' earns ' || v_emp.salary
        );
    END LOOP;
    CLOSE c_emp;               -- Step 4: ALWAYS close!
END;
/

-- ─── STYLE 3: Parameterized Cursor (reusable) ───
DECLARE
    CURSOR c_dept_emp (p_dept_id NUMBER) IS
        SELECT first_name, salary
        FROM   employees
        WHERE  dept_id = p_dept_id;
BEGIN
    DBMS_OUTPUT.PUT_LINE('── Dept 10 ──');
    FOR r IN c_dept_emp(10) LOOP
        DBMS_OUTPUT.PUT_LINE(r.first_name || ' - ' || r.salary);
    END LOOP;

    DBMS_OUTPUT.PUT_LINE('── Dept 20 ──');
    FOR r IN c_dept_emp(20) LOOP
        DBMS_OUTPUT.PUT_LINE(r.first_name || ' - ' || r.salary);
    END LOOP;
END;
/
💡 Performance Tip Prefer the Cursor FOR Loop (Style 1) for most cases — Oracle optimizes it internally. Explicit cursors are needed when you require finer control (e.g., UPDATE with WHERE CURRENT OF).

Interview Q Frequently Asked

"What is the difference between implicit and explicit cursors? When would you choose one over the other?"

▶ Show Answer
Implicit cursors are automatically created by Oracle for every DML statement and SELECT INTO. You access them via SQL%FOUND, SQL%ROWCOUNT etc. They're fine for single-row operations.

Explicit cursors are manually declared and used when your query returns multiple rows that need to be processed row-by-row. You have full control: OPEN, FETCH, CLOSE.

Choose explicit when: processing multiple rows, needing cursor attributes (%NOTFOUND, %ROWCOUNT), doing parameterized queries, or using FOR UPDATE / WHERE CURRENT OF for row-level locking.

Exception Handling

Graceful error management — what separates pros from beginners
🛡️ Analogy Exception handling is like an airbag in a car. In normal driving (BEGIN block), the airbag is inactive. When a crash happens (an error occurs), the airbag deploys (EXCEPTION block runs). Without it, the crash is fatal to your program. With it, you survive and can handle the situation gracefully.
PL/SQL — Exception Handling
DECLARE
    v_emp    employees%ROWTYPE;
    v_empid  NUMBER := 999;   -- Non-existent ID

    -- User-defined exception
    e_low_salary  EXCEPTION;
    v_salary      NUMBER;

BEGIN
    -- This will raise NO_DATA_FOUND
    SELECT * INTO v_emp
    FROM   employees
    WHERE  emp_id = v_empid;

    -- Manually RAISE user-defined exception
    v_salary := 25000;
    IF v_salary < 30000 THEN
        RAISE e_low_salary;
    END IF;

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        DBMS_OUTPUT.PUT_LINE('❌ Employee ID ' || v_empid || ' not found.');

    WHEN TOO_MANY_ROWS THEN
        DBMS_OUTPUT.PUT_LINE('❌ Query returned more than 1 row!');

    WHEN e_low_salary THEN
        DBMS_OUTPUT.PUT_LINE('❌ Salary too low: ' || v_salary);

    WHEN OTHERS THEN
        -- SQLCODE: Oracle error number (negative)
        -- SQLERRM: Error message text
        DBMS_OUTPUT.PUT_LINE('❌ Error: ' || SQLCODE || ' - ' || SQLERRM);
        RAISE;   -- Re-raise to propagate upward
END;
/

3.3

Stored Procedures

Named, reusable PL/SQL programs stored in the database
🔧 Analogy A stored procedure is like a saved macro in the database. Instead of writing the same 50-line PL/SQL block every time, you write it once, save it with a name, and call it with just one line. Like pressing a button on a remote to perform a complex sequence.
PL/SQL — Stored Procedures (IN, OUT, IN OUT)
-- ─── CREATE a stored procedure ───
CREATE OR REPLACE PROCEDURE give_raise (
    p_emp_id    IN     employees.emp_id%TYPE,    -- Input
    p_pct       IN     NUMBER,                   -- Input (percentage)
    p_new_sal   OUT    employees.salary%TYPE,    -- Output (returned)
    p_status    OUT    VARCHAR2                  -- Output message
)
AS
    v_curr_sal  employees.salary%TYPE;
BEGIN
    SELECT salary INTO v_curr_sal
    FROM   employees
    WHERE  emp_id = p_emp_id;

    p_new_sal := v_curr_sal * (1 + p_pct / 100);

    UPDATE employees
    SET    salary = p_new_sal
    WHERE  emp_id = p_emp_id;

    p_status := 'SUCCESS: Salary updated to ' || p_new_sal;
    COMMIT;

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        p_status := 'ERROR: Employee ' || p_emp_id || ' not found';
    WHEN OTHERS THEN
        p_status := 'ERROR: ' || SQLERRM;
        ROLLBACK;
END give_raise;
/

-- ─── CALL the procedure ───
DECLARE
    v_new_salary  employees.salary%TYPE;
    v_msg         VARCHAR2(200);
BEGIN
    give_raise(
        p_emp_id  => 1,
        p_pct     => 15,          -- 15% raise
        p_new_sal => v_new_salary,
        p_status  => v_msg
    );
    DBMS_OUTPUT.PUT_LINE(v_msg);
    DBMS_OUTPUT.PUT_LINE('New Salary: ' || v_new_salary);

-- ─── CREATE a function ───
CREATE OR REPLACE FUNCTION get_annual_salary (
    p_emp_id  IN  employees.emp_id%TYPE
) RETURN NUMBER
IS
    v_monthly  employees.salary%TYPE;
BEGIN
    SELECT salary INTO v_monthly
    FROM   employees
    WHERE  emp_id = p_emp_id;

    RETURN v_monthly * 12;   -- MUST return something

EXCEPTION
    WHEN NO_DATA_FOUND THEN
        RETURN 0;   -- Return 0 if employee not found
END get_annual_salary;
/

END;

3.5

Triggers

Automatic code that fires on database events
🔔 Analogy A trigger is like a motion sensor light. You don't turn the light on manually — it automatically turns on when it detects motion (an event). In the database, the "motion" is an INSERT, UPDATE, or DELETE on a table, and the "light" is your PL/SQL code that runs automatically.
PL/SQL — Triggers
-- ─── 1. BEFORE INSERT trigger: Auto-generate emp_id ───
CREATE OR REPLACE TRIGGER trg_emp_id
BEFORE INSERT ON employees
FOR EACH ROW                       -- Row-level: fires per row
BEGIN
    IF :NEW.emp_id IS NULL THEN
        SELECT emp_seq.NEXTVAL   -- From a SEQUENCE
        INTO   :NEW.emp_id
        FROM   DUAL;
    END IF;
    -- :NEW refers to the new row being inserted
    :NEW.hire_date := SYSDATE;   -- Auto-set hire date
END;
/

-- ─── 2. AFTER UPDATE trigger: Audit log ───
CREATE TABLE salary_audit (
    audit_id    NUMBER        GENERATED ALWAYS AS IDENTITY,
    emp_id      NUMBER,
    old_salary  NUMBER,
    new_salary  NUMBER,
    changed_by  VARCHAR2(50),
    changed_at  TIMESTAMP     DEFAULT SYSTIMESTAMP
);

CREATE OR REPLACE TRIGGER trg_salary_audit
AFTER UPDATE OF salary ON employees
FOR EACH ROW
WHEN (:NEW.salary != :OLD.salary)   -- Only if actually changed
BEGIN
    INSERT INTO salary_audit (emp_id, old_salary, new_salary, changed_by)
    VALUES (:NEW.emp_id, :OLD.salary, :NEW.salary, USER);
    -- :OLD = previous value, :NEW = new value
    -- USER = current logged-in DB user
END;
/

-- ─── 3. INSTEAD OF trigger (on a VIEW) ───
CREATE VIEW emp_dept_view AS
    SELECT e.emp_id, e.first_name, d.dept_name
    FROM   employees e JOIN departments d ON e.dept_id = d.dept_id;

CREATE OR REPLACE TRIGGER trg_instead_emp
INSTEAD OF INSERT ON emp_dept_view
FOR EACH ROW
BEGIN
    -- Redirect INSERT on view to the base table
    INSERT INTO employees (emp_id, first_name)
    VALUES (:NEW.emp_id, :NEW.first_name);
END;
/

3.6

Packages

Grouping related code — the professional way
📦 Analogy A package is like a toolbox. Instead of having tools (procedures, functions, variables) scattered everywhere, you organize them into themed toolboxes: one for employee operations, one for payroll, one for reporting. The Package Spec is the label on the toolbox (what's inside), and the Package Body is the actual tools with all their mechanics.
PL/SQL — Package Spec + Body
-- ─── PACKAGE SPECIFICATION (the "interface") ───
-- Only PUBLIC items declared here are visible to the outside
CREATE OR REPLACE PACKAGE emp_mgmt AS

    -- Public constants
    c_min_salary  CONSTANT NUMBER := 30000;

    -- Public type declaration
    TYPE t_emp_rec IS RECORD (
        emp_id   employees.emp_id%TYPE,
        name     VARCHAR2(100),
        salary   employees.salary%TYPE
    );

    -- Public procedure signatures (no body here)
    PROCEDURE hire_employee (
        p_name    IN  VARCHAR2,
        p_salary  IN  NUMBER,
        p_dept    IN  NUMBER,
        p_emp_id  OUT NUMBER
    );

    FUNCTION get_salary (p_emp_id IN NUMBER) RETURN NUMBER;

    PROCEDURE terminate_employee (p_emp_id IN NUMBER);

END emp_mgmt;
/

-- ─── PACKAGE BODY (the "implementation") ───
CREATE OR REPLACE PACKAGE BODY emp_mgmt AS

    -- Private variable (only accessible within this package)
    g_last_hire_id  NUMBER := 0;

    -- Private helper (not declared in spec)
    FUNCTION next_emp_id RETURN NUMBER IS
        v_max NUMBER;
    BEGIN
        SELECT NVL(MAX(emp_id),0)+1 INTO v_max FROM employees;
        RETURN v_max;
    END;

    -- IMPLEMENT hire_employee
    PROCEDURE hire_employee (
        p_name    IN  VARCHAR2,
        p_salary  IN  NUMBER,
        p_dept    IN  NUMBER,
        p_emp_id  OUT NUMBER
    ) IS
    BEGIN
        IF p_salary < c_min_salary THEN
            RAISE_APPLICATION_ERROR(-20001, 'Salary below minimum!');
        END IF;

        p_emp_id := next_emp_id();
        INSERT INTO employees (emp_id, first_name, salary, dept_id)
        VALUES (p_emp_id, p_name, p_salary, p_dept);
        g_last_hire_id := p_emp_id;
        COMMIT;
    END hire_employee;

    -- IMPLEMENT get_salary
    FUNCTION get_salary (p_emp_id IN NUMBER) RETURN NUMBER IS
        v_sal NUMBER;
    BEGIN
        SELECT salary INTO v_sal
        FROM   employees WHERE emp_id = p_emp_id;
        RETURN v_sal;
    EXCEPTION
        WHEN NO_DATA_FOUND THEN RETURN NULL;
    END get_salary;

    PROCEDURE terminate_employee (p_emp_id IN NUMBER) IS
    BEGIN
        DELETE FROM employees WHERE emp_id = p_emp_id;
        COMMIT;
    END terminate_employee;

END emp_mgmt;
/

-- ─── USING the package ───
BEGIN
    DECLARE
        v_id  NUMBER;
    BEGIN
        emp_mgmt.hire_employee('Sneha', 75000, 10, v_id);
        DBMS_OUTPUT.PUT_LINE('Hired. ID = ' || v_id);
        DBMS_OUTPUT.PUT_LINE('Salary: ' || emp_mgmt.get_salary(v_id));
    END;
END;
/
4.1

Collections & Records

Arrays and structs in PL/SQL
PL/SQL — Collections
DECLARE
    -- ─── 1. VARRAY (fixed max size) ───
    TYPE t_skills IS VARRAY(5) OF VARCHAR2(50);
    v_skills  t_skills := t_skills('SQL', 'PL/SQL', 'Java');

    -- ─── 2. Nested Table (unbounded array) ───
    TYPE t_names IS TABLE OF VARCHAR2(100);
    v_names   t_names := t_names();  -- Empty, extend before use

    -- ─── 3. Associative Array (key-value, most flexible) ───
    TYPE t_sal_map IS TABLE OF NUMBER INDEX BY VARCHAR2(50);
    v_salaries  t_sal_map;

    -- ─── 4. Record type ───
    TYPE t_emp_info IS RECORD (
        emp_id    NUMBER,
        name      VARCHAR2(100),
        salary    NUMBER,
        is_active BOOLEAN := TRUE
    );
    v_emp  t_emp_info;

BEGIN
    -- Using VARRAY
    FOR i IN 1..v_skills.COUNT LOOP
        DBMS_OUTPUT.PUT_LINE('Skill: ' || v_skills(i));
    END LOOP;

    -- Using Nested Table
    v_names.EXTEND(3);   -- Add 3 slots
    v_names(1) := 'Alice';
    v_names(2) := 'Bob';
    v_names(3) := 'Carol';

    -- Using Associative Array
    v_salaries('Engineering') := 95000;
    v_salaries('Finance')     := 85000;
    DBMS_OUTPUT.PUT_LINE('Eng: ' || v_salaries('Engineering'));

    -- Using Record
    v_emp.emp_id := 1;
    v_emp.name   := 'Arjun';
    v_emp.salary := 78000;
    DBMS_OUTPUT.PUT_LINE(v_emp.name || ' earns ' || v_emp.salary);
END;
/
4.2

Bulk Operations — BULK COLLECT & FORALL

Processing thousands of rows efficiently
🚛 Analogy Without BULK: Moving furniture one item per trip from floor 5 to a truck — each trip has overhead (elevator call, loading, unloading). With BULK COLLECT/FORALL: Load the entire floor's contents onto a cart, one trip to the truck. Dramatically fewer context switches between PL/SQL and SQL engines.
PL/SQL — BULK COLLECT & FORALL
DECLARE
    TYPE t_emp_ids   IS TABLE OF employees.emp_id%TYPE;
    TYPE t_salaries  IS TABLE OF employees.salary%TYPE;
    TYPE t_emp_rows  IS TABLE OF employees%ROWTYPE;

    v_ids      t_emp_ids;
    v_sals     t_salaries;
    v_emps     t_emp_rows;

BEGIN
    -- BULK COLLECT: Fetch ALL rows at once into a collection
    SELECT emp_id, salary
    BULK COLLECT INTO v_ids, v_sals
    FROM   employees
    WHERE  dept_id = 10;

    DBMS_OUTPUT.PUT_LINE('Fetched: ' || v_ids.COUNT || ' rows');

    -- LIMIT: Don't load ALL rows into memory at once!
    SELECT * BULK COLLECT INTO v_emps
    FROM   employees
    FETCH FIRST 100 ROWS ONLY;  -- Or use cursor FETCH LIMIT

    -- FORALL: Batch DML (much faster than row-by-row loop!)
    -- Apply 10% raise to all fetched employees in ONE SQL statement
    FORALL i IN 1..v_ids.COUNT
        UPDATE employees
        SET    salary = v_sals(i) * 1.10
        WHERE  emp_id = v_ids(i);

    -- Check how many rows were updated
    DBMS_OUTPUT.PUT_LINE('Updated: ' || SQL%ROWCOUNT || ' rows');
    COMMIT;

END;
/
⚡ Performance Impact FORALL can be 10x–100x faster than a row-by-row loop for large datasets. Every row-by-row loop causes a "context switch" between PL/SQL and SQL engines. FORALL makes one round trip for thousands of rows.
4.3

Dynamic SQL — EXECUTE IMMEDIATE

Building and running SQL at runtime
🔧 Analogy Normal SQL is like ordering a pre-set meal from a fixed menu. Dynamic SQL is like giving the chef a handwritten note at runtime saying exactly what ingredients and cooking method you want — you construct the instruction on the fly based on conditions you only know at runtime.
PL/SQL — Dynamic SQL
DECLARE
    v_table    VARCHAR2(50) := 'employees';
    v_sql      VARCHAR2(1000);
    v_count    NUMBER;
    v_dept     NUMBER := 10;
    v_sal      employees.salary%TYPE;

BEGIN
    -- 1. Simple DDL via EXECUTE IMMEDIATE
    EXECUTE IMMEDIATE 'CREATE TABLE temp_log (msg VARCHAR2(200), ts TIMESTAMP DEFAULT SYSTIMESTAMP)';

    -- 2. Dynamic query with INTO (single row)
    v_sql := 'SELECT COUNT(*) FROM ' || v_table;
    EXECUTE IMMEDIATE v_sql INTO v_count;
    DBMS_OUTPUT.PUT_LINE('Count: ' || v_count);

    -- 3. BIND VARIABLES (CRITICAL for security & performance!)
    -- Never concatenate user input! Use USING instead.
    v_sql := 'SELECT MAX(salary) FROM employees WHERE dept_id = :dept';
    EXECUTE IMMEDIATE v_sql INTO v_sal USING v_dept;
    DBMS_OUTPUT.PUT_LINE('Max Sal Dept '||v_dept||': '||v_sal);

    -- 4. Dynamic DML
    EXECUTE IMMEDIATE
        'UPDATE employees SET salary = salary * :pct WHERE dept_id = :dept'
        USING 1.05, v_dept;

    DBMS_OUTPUT.PUT_LINE('Rows updated: ' || SQL%ROWCOUNT);
    COMMIT;

EXCEPTION
    WHEN OTHERS THEN
        DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
        ROLLBACK;
END;
/
🚨 Security Critical: SQL Injection NEVER build dynamic SQL by concatenating user input like this: 'WHERE name = ''' || user_input || ''''. This opens your database to SQL injection attacks. ALWAYS use bind variables with :placeholder and USING clause.
4.4

Performance Optimization

Writing fast PL/SQL — think at scale

The Golden Rules of PL/SQL Performance

  • 🔴 Rule 1: Minimize context switches between SQL and PL/SQL engines — use BULK COLLECT and FORALL
  • 🟡 Rule 2: Write efficient SQL — indexes, avoid full table scans on large tables
  • 🟢 Rule 3: Never do row-by-row what SQL can do set-based in one statement
  • 🔵 Rule 4: Use bind variables — avoid hard parsing repeated SQL
  • 🟣 Rule 5: Limit COMMIT frequency — committing inside loops is extremely slow
PL/SQL — Performance Patterns
-- ❌ SLOW: Row-by-row update (1000 employees = 1000 round trips!)
FOR r IN (SELECT emp_id FROM employees WHERE dept_id=10) LOOP
    UPDATE employees SET salary=salary*1.1 WHERE emp_id=r.emp_id;
END LOOP;
COMMIT;

-- ✅ FAST: Single SQL statement (1 round trip!)
UPDATE employees SET salary = salary * 1.1 WHERE dept_id = 10;
COMMIT;

-- ❌ SLOW: COMMIT inside loop
FOR i IN 1..10000 LOOP
    INSERT INTO log_table VALUES(i);
    COMMIT;   -- 10000 COMMITS! Very slow.
END LOOP;

-- ✅ FAST: Single COMMIT after all inserts
DECLARE
    TYPE t_nums IS TABLE OF NUMBER;
    v_nums t_nums := t_nums();
BEGIN
    v_nums.EXTEND(10000);
    FOR i IN 1..10000 LOOP
        v_nums(i) := i;
    END LOOP;
    FORALL i IN 1..v_nums.COUNT
        INSERT INTO log_table VALUES(v_nums(i));
    COMMIT;  -- 1 COMMIT
END;

EXPLAIN PLAN FOR
SELECT * FROM employees WHERE salary > 80000;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

Interview Preparation

Top 20 questions asked in Oracle/PL/SQL interviews

Interview Q Q1 — Frequently Asked

"Explain the difference between a stored procedure and a function."

▶ Show Answer
A function must return exactly one value and can be called inside SQL statements (SELECT, WHERE). A procedure does not have to return a value (uses OUT parameters instead) and cannot be used in SQL expressions. Use functions for computations, procedures for processes/business logic that perform DML.

Interview Q Q2

"What is a cursor and why do we need it? What are cursor attributes?"

▶ Show Answer
A cursor is a pointer to the result set of a SQL query. We need it because PL/SQL's SELECT INTO only works for single-row results. For multi-row results, a cursor lets us iterate one row at a time. Cursor attributes: %FOUND (last FETCH returned a row), %NOTFOUND (no more rows), %ROWCOUNT (rows fetched so far), %ISOPEN (cursor is open).

Interview Q Q3

"What are the types of triggers? When would you NOT use a trigger?"

▶ Show Answer
Types: BEFORE/AFTER × INSERT/UPDATE/DELETE (row-level or statement-level), INSTEAD OF (on views), DDL triggers, Database event triggers (LOGON, STARTUP, etc.).

Avoid triggers for: complex business logic (use procedures instead), operations that could cause cascading triggers (mutating table errors), logic that needs to be testable independently, or when they would make debugging difficult.

Interview Q Q4

"Explain BULK COLLECT and FORALL. Give a real use case."

▶ Show Answer
BULK COLLECT retrieves multiple rows into a collection in a single round trip from SQL to PL/SQL engine. FORALL sends a batch DML operation (all array elements) in a single round trip from PL/SQL to SQL engine.

Use case: Year-end salary processing — BULK COLLECT all 50,000 employees, apply tiered bonus logic in PL/SQL, then FORALL UPDATE back to database. Instead of 50,000 round trips, you get 2 (one BULK COLLECT, one FORALL UPDATE).

Interview Q Q5

"What is the difference between TRUNCATE and DELETE? Can you ROLLBACK a TRUNCATE?"

▶ Show Answer
DELETE is DML — it removes rows one at a time, generates undo logs, can have a WHERE clause, fires triggers, and can be ROLLBACKed.
TRUNCATE is DDL — it deallocates all data pages, is much faster, cannot have WHERE clause, does not fire row-level triggers, and auto-COMMITs (cannot be rolled back). TRUNCATE also resets the high-water mark.
